How it works

A verifier between the model and your board

Every candidate design is checked by a deterministic electrical rule checker before it can reach you. If the model cannot reach a passing design within its retry budget, Ohmatic returns a clarifying question instead, and the broken draft stays in the box.

Normalize Generate Verify Deliver
⟲ correction loop · Verify back to Generate
  1. NormalizeA fine-tuned T5 rewrites any phrasing onto the model's trained input, then re-attaches the specifics it dropped.
  2. GenerateA fine-tuned Qwen3-8B emits a two-stage circuit: topology first, then spatial layout.
  3. VerifyThe ERC engine checks connectivity, power integrity, pin legality, and structure.
  4. CorrectOn a failure the model receives the findings and repairs its own design, up to three rounds.
  5. RefuseOut of retries, it returns a clarifying question. The broken candidate stays internal.
Architecture

Five stages, wrapped in a local service layer

Normalize, Generate, Verify, Correct, Refuse. An amber self-correction loop feeds the rule-checker findings back to the model; a red killswitch returns a clarifying question when the retries run out.

Benchmark

An 8B fine-tune. Zero broken boards delivered.

62 single-circuit tasks from PCBBench (PCBSchemaGen, MIT), a third-party suite of someone else's prompts, run end to end through the full pipeline and scored by Ohmatic's own deterministic ERC engine, the same checker that gates production.

Ohmatic's broken rate, 95% upper bound

Identical prompts and identical ERC verifier for every leg; the competitor gets the schema and component registry but not the ERC rules (condition C1), so the checker can't be gamed. Codex ran single-shot via its CLI at xhigh reasoning effort. PCBBench is harder and out-of-distribution, so Ohmatic's clean rate is lower and it abstains on about half. But where the frontier model answers every task and ships 22 broken boards, Ohmatic asks you to clarify and ships none across all three precisions (q4, q8, bf16). Ohmatic is trained to ≤30-component circuits while PCBBench spans up to 50, so its largest tasks are out-of-scope and it abstains on them. Full methodology →

Parts

A bill of materials, with disclosed links

Once a design is verified, Ohmatic can build a parts list and offer optional buy-links to component distributors. DigiKey is live today; the rest are coming soon. The schematic itself stays exactly as it was verified.

  • Buy-links are search or deep-links you open in your own browser.
  • Where a link is an affiliate link, a commission disclosure is shown beside it.
  • No storefront scraping, no carts filled on your behalf, no undisclosed links.

Local-first

Runs on your machine

No account, no telemetry. The whole pipeline runs on your machine, with no cloud calls in the core loop, so your prompts and designs stay on your device. The only thing that leaves is an optional parts lookup you choose to open, handled by your browser like any other link. 

# clone and boot the local stack
git clone https://github.com/VittoriaLanzo/Ohmatic.git && cd Ohmatic
ohmatic start   # first run scans your hardware and offers the matching model
